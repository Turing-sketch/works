## 一、AI 交互日志

本报告记录本项目实际的人机协作过程。AI 被用于要求拆解、代码草拟、报错定位和文档整理；数据选择、加分项范围、是否接入 API 等决定由本人确认。对于模型与数据结论，均以真实运行输出为准，而不是直接采用语言模型给出的数字。

### 1. 任务拆解与方案选择

**关键 Prompt：**“分析期末作业要求，按满分标准列出交付物和加分项；个人信息留空。”  
**AI 输出摘要：**将任务拆为 M1 数据处理、M2 四项分析、M3 两类模型、M4 问答接口、Gradio、地图、README、运行说明 PDF 和仓库。  
**采用与修改：**采用模块化结构，但拒绝使用需要付费或密钥的大模型 API。最终选择“规则匹配＋关键词提取＋数字抽取”的本地问答方案，并同时完成 Gradio 和地图两个加分项。

### 2. M1 数据质量与清洗

**关键 Prompt：**“不要只用 dropna，请根据出租车业务含义制定可解释的异常规则，并生成字段级质量报告。”  
**AI 输出摘要：**建议检查时间倒置、非本月上车、零/极端里程、负/极端金额、非法区域和乘客数缺失。  
**采用与修改：**采用时间 1–180 分钟、距离 0.1–100 英里、基础车费 0–500 美元、总金额 0–1000 美元等边界。乘客数缺失填 1，但没有让它参与需求预测，避免填补值放大模型偏差。新增 `speed_mph` 和 `fare_per_mile` 两个衍生特征。

### 3. AI 犯错案例：中文字体缺失

**现象：**AI 初始代码设置了常见中文字体名称，但运行时产生大量 “Glyph missing” 警告，图片中的中文可能显示为方框。  
**如何发现：**查看真实运行日志，而不是只检查代码能否结束；随后打开输出图进行视觉核验。  
**修正：**在项目中附带 Noto Sans SC 字体文件，运行时通过 `font_manager.addfont()` 显式注册，再重新生成所有图片。这个案例说明“配置了字体名”并不等于运行环境真的存在该字体。

### 4. AI 犯错案例：随机划分可能造成时间泄漏

**现象：**最初方案按 8:2 随机切分聚合样本。出租车需求具有明显时间周期，同一天相邻小时会高度相似，随机切分会让未来日期的信息进入训练集。  
**如何发现：**检查样本定义和拆分逻辑，发现随机切分虽满足字面比例，却不符合预测任务。  
**修正：**改为按日期排序，前 80% 日期训练、后 20% 日期测试；同时固定 Python、NumPy、PyTorch 与随机森林的随机种子。这样指标可能不如随机划分漂亮，但更可信。

### 5. M2 分析与可视化

**关键 Prompt：**“四类分析必须各有图，中文标题完整；自选洞察不能只是重复画订单量。”  
**AI 输出摘要：**工作日/周末小时曲线、热门区域条形图＋热力图、距离—车费散点图＋小时车费、速度—小费率双轴图，并用分级设色地图呈现空间需求。  
**采用与修改：**采用上述设计。散点图固定随机种子抽样 3 万条，避免 348 万点重叠和绘图卡顿；聚合统计仍使用全部清洗后数据。自选洞察将交通效率与消费行为放在同一时间轴上，便于识别拥堵与小费变化。

### 6. M3 模型与实验

**关键 Prompt：**“以区域—日期—小时为样本，说明特征原因，用 PyTorch 神经网络和随机森林做可复现对比。”  
**AI 输出摘要：**区域 ID 独热编码，小时、星期、周末、高峰与日期序号标准化；神经网络为 64—32 隐藏层；随机森林 180 棵树。  
**采用与修改：**选取最活跃的 80 个区域，减少极稀疏区域产生的零膨胀。对比结果以 `m3_model_metrics.csv` 为准。随机森林在当前表格型、小样本聚合任务上通常更稳；神经网络训练和扩展性较好，但对结构、学习率与轮数更敏感。

### 7. M4 问答与界面

**关键 Prompt：**“没有大模型 API，仍需支持至少五类自然语言问题，并返回结论、解释、文件路径。”  
**AI 输出摘要：**通过关键词、正则数字抽取和优先级规则识别高峰、区域排名、周末对比、费用估算、需求预测、拥堵六类意图。  
**采用与修改：**加入未识别问题的示例提示；费用估算明确标注为历史中位数而非实时报价，防止误导。命令行和 Gradio 共用同一个 `TaxiQA` 类，避免两套逻辑不一致。

## 二、Native、Prompt 与 Vibe 三阶段对比

选取“查询需求最高小时”这一典型功能进行比较。

### Native 版（独立编写）

```python
hour_count = df.groupby("hour").size()
print(hour_count.idxmax(), hour_count.max())
```

优点是代码短、逻辑完全可控；缺点是只能执行固定查询，不理解用户表达，也没有解释和文件路径。适合验证核心统计是否正确。

### Prompt 版（AI 辅助）

```python
def peak_hour(df):
    counts = df.groupby("hour").size()
    h = int(counts.idxmax())
    return {"hour": h, "count": int(counts.max()),
            "chart": "outputs/m2_1_demand.png"}
```

AI 帮助把一次性脚本封装为可测试函数，效率明显提高。不过仍需人工检查 `hour` 是否已经从上车时间正确提取、数据是否完成清洗，以及路径是否为相对路径。理解深度取决于使用者是否能解释每一步。

### Vibe 版（自然语言对话驱动）

```python
if any(k in question for k in ["最高峰", "最忙", "高峰时段"]):
    g = df.groupby("hour").size()
    h = int(g.idxmax())
    return f"需求最高的是 {h}:00–{h+1}:00，共 {g.max():,} 单。"
```

Vibe 版从“用户会怎么问”出发，通过多轮对话逐步加入同义词、格式化结论和图表路径，原型效率最高，也最接近产品体验。但它容易产生规则冲突或遗漏边界，例如问题同时包含“周末”和“最忙”时，意图优先级会影响结果。因此必须准备测试问题并逐条验证。

综合来看，Native 版最有利于建立基础理解；Prompt 版适合快速形成工程结构；Vibe 版适合探索交互，但最依赖人工验收。最佳方式不是三选一，而是先用 Native 验证统计逻辑，再用 Prompt 工程化，最后用 Vibe 完成交互。

## 三、反思

完成本次作业后，我更清楚地认识到，AI 擅长快速生成项目骨架、列出可能的异常规则、补充注释并解释报错，但它不能替代对数据语境的判断。比如随机 8:2 拆分表面上符合要求，却可能造成时间泄漏；设置一个中文字体名称也不代表运行环境真正安装了该字体。只有查看真实日志、指标和图片，问题才会暴露。AI 还能很快给出“看起来合理”的数值或结论，因此所有统计必须由程序从真实数据生成，不能直接复制回答。对我而言，最有效的协作方式是把 AI 当作速度很快的结对程序员：让它提出方案和候选实现，由我决定目标、检查边界、运行测试并对最终结果负责。AI 的能力边界并不只是“会不会写代码”，更在于它不了解本地环境、评分语境和未明说的业务后果。今后我会保留关键 Prompt 和修改决策，优先验证数据、路径、随机性与可复现性，而不是只追求代码量或界面效果。
